## 2026-06-03 follow-up — forwarded contact-form → new thread

- **Plan:** `docs/plans/2026-06-03-inbox-forwarded-form-new-thread.md`

**Problem fixed:** a website contact-form submission forwarded into a connected mailbox (sender ∈ team forwarders, or a known form platform) used to auto-draft a reply **glued to the forwarder's thread** with a "Re: &lt;form subject&gt;" subject. It now starts a **clean NEW thread to the actual client** with a fresh subject (`Thanks for reaching out`), and the draft stays reconcilable by the learning loop.

**How:** `maybeAutoGenerateDraft(email, connection, opportunityId, submitter?)` — when `submitter` (the parsed `contactFormSubmitter`) is present it takes a new-thread branch: `AIDraftService.generateDraft` **new-email path** (recipient = submitter, no inbound thread, grounded in the form message) → `placeNewThreadDraft` (`src/lib/api/services/mailbox-draft-push.ts`) → `provider.createNewThreadDraft` (no threadId; Gmail `message.threadId` / M365 `conversationId` captured) → stores the **new** thread id on `ai_draft_history.thread_id`, links it via `opportunity_email_threads`, idempotency keyed on `(connection_id, opportunity_id)`. Never auto-sends (cold first contact is review-only). The `create_new` lead branch (brand-new contact-form client, previously un-drafted) now drafts too, gated to `contactFormSubmitter`. **Category-gate fix:** contact forms classify as `general` — the forwarder subject "New submission" used to match `sub` → `subtrade_coordination` → `draft_on_request`, silently suppressing the draft. The manual Pipeline **Draft** route does the same (detects via `extractContactFormSubmission` on the latest inbound activity).

**Reconciliation correctness:** because the new thread is linked, the user's send creates an outbound activity on it and `reconcilePendingMailboxDrafts` (keyed on `thread_id` = the new thread) classifies it `used → sent_from_mailbox`. Without the `thread_id` stamp + link the send would be invisible and the draft would rot to `discarded_in_mailbox` after TTL — locked by `draft-reconciliation.test.ts` ("new-thread (contact-form)").

**Verified:** `tsc --noEmit` clean; inbox suite green incl. new cases in `auto-draft-mailbox.test.ts`, `mailbox-draft-push.test.ts`, `draft-reconciliation.test.ts`, and `tests/unit/api/email-draft-route.test.ts` (manual path). **No migration** — `auto_drafted` already in the live `ai_draft_history_status_check`.

**Live acceptance (run before broad rollout):** forward a real website contact-form submission into a connected `phase_c` mailbox → confirm the draft is a **new thread** to the client (not the forwarder), subject `Thanks for reaching out` (not "Re:"), lands in Gmail/Outlook Drafts; edit + send; confirm the next sync reconciles `status → sent_from_mailbox`. Repeat via the manual Pipeline **Draft** button. Both providers.
