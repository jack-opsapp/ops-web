## Status and verdict

The five production symptoms are related, but they are not one bug. They came from five broken boundaries:

1. provider discovery folders were treated as message authorship when assigning direction;
2. provider thread identity was treated as customer/lead identity;
3. AI failures and invalid output were converted into a valid `new_lead` result, while some persistence errors did not stop cursor advancement;
4. contact enrichment was fill-first and its provenance writers did not consistently match the live schema;
5. project/opportunity mirrors and the won transition were maintained by calling convention instead of a database invariant.

The current isolated hardening diff fixes the confirmed Gmail/parser/lifecycle/project defects prospectively and adds focused regression coverage. It also closes the core browser-route tenant/permission gaps found around service-role email operations, binds Gmail/Microsoft OAuth initiation to one authenticated company/user plus one short-lived callback nonce, isolates connection identity by provider, and binds alert reconnects to the exact failed connection/mailbox. It does **not** mutate Canpro data, apply any migration, run the review backfill, split Sandra Dunford's opportunity, commit, push, or deploy.

Eight executable expand migrations and one held post-deploy contract SQL artifact are required by the application changes. They have only static SQL contract coverage in this workspace; no real PostgreSQL trigger, RLS, grant, concurrency, or rollback execution has been performed. Migrations `2000` through `2100` are the compatible expand phase and must verify before the application deploy. Contract artifact `2050` is deliberately outside `supabase/migrations`, so an apply-all command cannot run it before the compatible application is live and verified.

## A. Ingestion completeness

### Gmail current path

- Gmail watch/Pub/Sub is a wake-up signal. Gmail History is the incremental source of truth; the scheduled cron and manual sync call the same `SyncEngine.runSync()` path.
- `GmailProvider.fetchNewEmailsSince()` makes one unfiltered History request chain for the mailbox, follows every `nextPageToken`, deduplicates provider message IDs, fetches every materialized message, and throws typed errors on any page or message failure. Gmail `fetchSentEmailsSince()` is not called by steady sync; direction is resolved after the unified discovery pass.
- `GmailProvider.searchEmails()` now paginates to mailbox exhaustion or the requested unique-message limit. Historical import paginates message listing, fetches full MIME bodies, and fails instead of silently accepting a truncated bound or individual fetch/write failure.
- A new Gmail connection with no cursor reconciles from `last_synced_at` when available, otherwise `connection.createdAt`, before persisting a fresh boundary. An expired History ID fixes a 15-minute-overlap anchor and snapshots a fresh target History ID, then durably stores both before listing. Each successful cycle processes a bounded batch of up to 10 listing pages, stopping once it reaches at least 500 unique threads, and persists the next provider page token only after every fetched message and semantic side effect in that batch succeeds. Later cycles resume that exact page; the old cursor remains authoritative until the final page succeeds, when the fresh target is committed and all recovery fields clear. A crash/error replays the current batch idempotently instead of skipping the remainder.
- Activity, thread, provenance, correspondence event/projection, deterministic stage, AI classification/stage/summary, and canonical accept/project failures now reject the cycle. Sync classification requires one exact known result for every input; transport, parse, empty, partial, duplicate, and unknown-result failures throw rather than becoming synthetic `skip` rows. Future provider-backed activities use connection-scoped `(company_id, email_connection_id, email_message_id)` identity. Duplicate provider messages replay the remaining idempotent semantic side effects rather than treating an activity row as proof of full completion. Wizard imports now persist every exact analyzed provider message instead of one thread-level summary shell; an ordinary payload without complete identities stops and requires reanalysis.
- The cron now returns HTTP 503 when `SyncCycleResult.errors` is non-empty or the stale-lead sweep fails, so scheduler/monitoring infrastructure can observe a failed semantic cycle.
- The staleness heartbeat now records an alert only when at least one configured delivery channel actually succeeds. If notification and email delivery both fail or are unavailable, it writes no false dedupe receipt and returns HTTP 503 with the delivery failures, so the next run can retry.

OAuth connection setup now has the same tenant boundary. Gmail and Microsoft 365 initiation require a verified operator with `settings.integrations` permission whose authenticated company and user exactly match the request. The browser receives a 256-bit opaque nonce with a 10-minute lifetime; only its SHA-256 digest and server-verified provider/company/user/type/source context are stored. The callback atomically consumes the provider-bound unexpired row with `DELETE ... RETURNING`, then re-verifies that the returning OPS cookie resolves to that exact permitted company/user before token exchange. That second browser-session binding prevents a relayed provider-consent URL from attaching a victim's mailbox to the initiator's tenant. Concurrent/replayed callbacks, the wrong provider, expired state, malformed state, a different/no OPS session, and the former unsigned tenant payload all fail closed before connection persistence. Wizard connects read and upsert only `(company_id, provider, email)`, so the same address on Gmail and Microsoft cannot share refresh tokens, cursors, webhook state, or settings. Connection-read errors fail closed instead of seeding defaults over an unknown row. Alert-email reconnects now send logged-out recipients through OPS login and back to the same confirmation; the link is verified against one exact active or reconnect-required, sync-enabled connection ID, company, provider, type, and normalized mailbox before tenant identity is displayed or OAuth state is minted. The callback requires the provider profile to match that mailbox, re-reads the same eligibility, and compare-and-set updates the exact email/status/sync-enabled snapshot. A stale link cannot reactivate a row that was disconnected before or during the callback.

### What is still missing

This is stronger retry safety, not a losslessness proof. There is no independent daily provider/database parity ledger or 48-72 hour repair sweep. `email_ingest_heartbeat_log` records stale alerts, not positive cycle receipts or provider-vs-OPS message counts. A provider anomaly, pre-existing gap, or message-level defect that never invalidates the cursor therefore has no independent detector.

Required follow-up design:

1. Run a daily, paginated 48-72 hour provider/database anti-join keyed by connection plus provider message ID.
2. Replay missing messages through the same normalized ingestion path; do not patch aggregates directly.
3. Persist positive cycle receipts: provider messages seen, OPS messages present, replayed, failed, old/new cursor, and unresolved parity count.
4. Add a durable retry/dead-letter record and persistent operator notification for any bounded recovery or semantic retry that cannot converge.
5. Keep heartbeat staleness alerts, but alert separately on any nonzero parity discrepancy.

Microsoft 365 is also prospectively stronger. Graph has no mailbox-wide message delta, so the provider now encodes one mailbox-folder delta plus one message delta/continuation per discovered live folder in a versioned `m365:v2` cursor. This covers Inbox, Sent, Archive, and rule/custom-folder delivery instead of assuming two folders are the mailbox. A durable pending-folder queue and advancing `@odata.nextLink` checkpoints share a 50-page cycle budget; a continuation is returned to SyncEngine only with the messages it represents and becomes durable only after semantic persistence succeeds. Legacy Inbox/Sent or raw cursors force a complete folder-inventory replay. Folder/message tombstones advance only their proven streams, unsent drafts are skipped, immutable provider IDs dedupe folder moves, and malformed/non-advancing pages fail closed. Full-thread fetches paginate or fail at their explicit cap. New Graph requests and subscription creation request `IdType="ImmutableId"` so folder moves do not change future provider identity.

Any environment that already has Microsoft 365 data has a compatibility gate: stored activities/events may contain mutable REST IDs, and existing subscriptions may have been created without the immutable-ID preference. Those subscriptions must be recreated and stored identities translated or reconciled under a reviewed transition; simply mixing old REST IDs with new immutable IDs can duplicate historical messages. The live production read found zero Microsoft 365 connections, so this is a future-provider rollout requirement here, not a current Canpro data blocker. New subscriptions use a random `clientState` secret; the database stores only its SHA-256 digest, and the webhook binds that digest plus Graph subscription ID to one active connection before service-role dispatch. This path has code/unit proof but no live Graph proof. The missing provider/database parity ledger still prevents a losslessness claim.

## B. Correct lead creation

### Wix/contact-form trace

1. The provider message is fetched in full and parsed by `extractContactFormSubmission()`; the platform/forwarding wrapper is not the customer.
2. `SENT` is checked before form parsing so an operator reply quoting an older form remains outbound. A genuine form notification is inbound and uses the parsed submitter's name/email/phone/address/message.
3. `buildLeadRoutingIdentity()` makes the submission message-scoped during analysis and persists `email:<provider>:<connection_id>:message:<message_id>` as its CRM identity, while preserving raw provider message/thread IDs on activity/audit fields.
4. Operator emails/domains/phones/addresses are removed from contact candidates. Real submitter/display-name evidence is accepted; a bare local part is not a verified name.
5. No raw provider-thread relationship is inherited or inserted for the form notification.

### Direct-customer trace

1. Ordinary correspondence keeps its raw provider thread as continuity identity.
2. Direction comes from author/label/operator identity, not the discovery bucket.
3. Inbound contact facts come from the external sender; outbound facts come from the external recipient. A real display name can enrich the lead; the local part alone cannot.
4. Exact thread inheritance is allowed for replies, then customer/client relationship matching applies.

The production Canpro connection's empty `userEmailAddresses` means an unlisted alias outside the connected address/company domain can still be misclassified until that identity set is populated.

## C. Lead updates on new data

New mail on an existing opportunity now runs canonical enrichment, writes/replays the immutable correspondence event, applies its counters/timestamps exactly once, evaluates deterministic stage rules, and then evaluates all eligible active threads through the AI reviewer. Summary writes and active-stage transitions are checked; a stage transition uses the locking service-role RPC so the opportunity and audit row cannot split. Manual and terminal stages are protected.

Provenance is consulted before contact replacement. Operator/confirmed fields win; verified form/sender evidence may replace legacy local-part/operator-signature contamination. The AI classifier receives the deterministic external customer identity, and its returned identity fields are never persisted; only stage and safe missing estimate/description facts are eligible. Missing model client identity does not discard an otherwise valid lead. Invalid model output no longer becomes `new_lead`, an incomplete/duplicated/unknown result rejects the cycle, and enrichment/sub-client failures occur before the durable message checkpoint so retry cannot skip them.

Cost caveat: evaluating every active target in batches of five removes the old first-20 blind spot but can increase OpenAI calls roughly in proportion to active threads. Resumable Gmail recovery can span multiple function cycles for a large gap, and Microsoft 365 recovery inventories the full folder tree and replays every live folder stream; both increase provider requests and function duration. The exact dollar impact is model-, mailbox-, and current-platform-price-dependent and was not calculated in this audit; it must be estimated against the deployed model/runtime before rollout. The inspected production project has no M365 connection today.

## D. Thread grouping

Future durable matching keys are scoped by provider and mailbox connection, then split by message type:

- ordinary conversation: `email:<provider>:<connection_id>:thread:<provider_thread_id>`;
- contact-form submission: `email:<provider>:<connection_id>:message:<provider_message_id>`;
- provider operations/audit: unchanged provider thread and message IDs.

This prevents two Wix customers in one provider thread from inheriting the same opportunity. Historical import now uses the same message-scoped route and full body parsing; the earlier report claim that it remained raw-thread-scoped is obsolete. Wizard continuation replaces preliminary scan subsets with the complete fetched messages, and merged ordinary leads retain/link every contributing raw provider thread. Ordinary provider-thread ownership is immutable after the first successful claim: all writers insert-if-absent and read the canonical winner instead of overwriting it. Manual send preflights a supplied reply thread before the irreversible provider send. If a verified same-company winner still appears while the provider request is in flight, the already-sent message adopts that owner for activity, event, projection, inbox cache, and lifecycle writes and returns success so auto-send cannot duplicate it. Other sync/import writers fail and retry before cursor advancement when a competing owner wins. Migration `2030` rejects a connection-backed thread link whose opportunity belongs to another company and rejects any later `opportunity_id` rewrite, including one attempted by an old application instance during rolling deploy.

Two residual boundaries remain:

- Existing unscoped `source_thread_key` and activity rows are deliberately not rewritten by this hardening. New writes are scoped, and legacy activity adoption requires connection proof; historical normalization still needs a separately reviewed compatibility backfill rather than guessed ownership.
- There is no atomic company + normalized-email client get-or-create constraint. The wizard's source-key collision path deterministically reuses the winning opportunity/client and guarded-merges only its request-created loser, but a crash after a racing client insert and before opportunity conflict recovery can still leave an orphan client.

Sandra is intentionally untouched. Fifteen raw threads are present, so neither one-thread-per-lead nor one-sender-per-lead is a safe historical rule. A reviewer must map each provider message/activity to an already-reviewed target opportunity/client before any move.

## E. Project linking and auto-win

The prepared database invariant makes the normalized contract explicit:

- `projects.opportunity_ref` and legacy `projects.opportunity_id` agree;
- `opportunities.project_ref` and `opportunities.project_id` agree;
- both sides point to the same active same-company rows;
- a direct linked project, or a previously linked approval project becoming `accepted`, `in_progress`, `completed`, or `closed`, wins the opportunity and writes one transition;
- the approval queue's deliberate `p_win_opportunity=false` insert remains non-winning until activation or a later canonical conversion;
- an already-linked conversion repairs all mirrors and still honors the requested win;
- soft/hard project deletion releases both reverse mirrors.

`ProjectService` now writes both project-side fields, and deterministic email acceptance calls the canonical conversion service with source `email_accept`. The service rejects an already-converted response without a project ID and preserves the RPC's `won` result.

The current Canpro readback found zero mirror disagreements, zero reverse-missing links among currently linked rows, and zero duplicate live projects. Mark and Derek are already repaired. The 20 in-progress projects without a resolvable opportunity link are not automatically defects; only accepted-estimate/source evidence or an operator-reviewed mapping can link them.

The fourth migration is deliberately additive. ## Safe backfill policy

Use `docs/backfills/2026-07-13-email-lead-project-repair-review.sql` through `psql` only. Its default flow creates session-local review tables, runs public-data discovery in a database-enforced `READ ONLY` transaction, and ends with `ROLLBACK`.

Durable mutation requires three independent gates:

1. `apply=true`;
2. an exact allowlist row with expected current values, reviewer, approval, and rationale;
3. `commit=true`; otherwise the rehearsal rolls back.

The script deliberately will not infer a link for an unlinked/manual project, auto-win an unlinked project, overwrite operator-confirmed contact data, infer direction without an activity allowlist, reassign an already-linked project, create target customers/opportunities for a split, or split Sandra by sender/subject/raw thread alone.

Approved activity moves/direction repairs recompute correspondence counts and last-message timestamps/direction from `opportunity_correspondence_events.occurred_at`, falling back to `activities.created_at` only when no matching event exists. They do not reconstruct historical correspondence classification, undo a prior lifecycle reset, or infer a new stage/summary/contact/title. The normal classifier/lifecycle path must be replayed and reviewed after an approved mapping commit.
