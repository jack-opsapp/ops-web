# Catalog Setup Wizard Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task. Phases live in linked per-phase files under ./2026-06-13-catalog-setup-wizard/ — execute in order P0 -> P6.

**Goal:** Stand up a trades company's entire OPS operating system (price book + optioned/tiered pricing, stock, trade/task types, estimate-line-item readiness) on OPS-Web via a guided, never-blocking wizard whose sources (QuickBooks, spreadsheet, document/photo, agent, template, manual) all feed one accept/edit/reject staging canvas committed in a single idempotent call.

**Architecture:** Conversation + live-building canvas — a full-page two-pane surface (driver left, canvas right); every proposed row is an accept/edit/reject card; commit via the merge-capable idempotent catalog_setup_save RPC. An always-on, suggest-only Setup Agent (owner approves every write) sits atop a deterministic floor (survey + CSV/XLSX mapper + per-trade template + manual); the deeper Phase C autonomous layer is gated.

**Tech Stack:** Next.js App Router, TypeScript, Supabase (Firebase-token bridge to service-role/RPC), TanStack Query, Zustand (persist), Radix, lucide-react, motion/react (Framer), vitest + @testing-library, Playwright (if configured), AI SDK / Anthropic SDK for the Setup Agent.

2. **[The Conversation + Live-Building Canvas (surface, state, modules)](2026-06-13-catalog-setup-wizard/01-the-conversation-live-building-canvas-surface-state-modules.md)** — Builds the wizard's pure foundation and shell: an in-memory staging-card data model + a TDD'd pure reducer/state machine (card states proposed/accepted/edited/rejected/merge; sources import/agent/template/manual; running totals), a persisted Zustand store mirroring useSetupStore (persist key, _hydrated gate, reset) so a refresh resumes the canvas, the full-page two-pane shell (left = driver pane: agent OR deterministic guided prompts; right = live-building canvas grouped by module), accept/edit/reject card components, a bespoke neutral-fill module-rail stepper (SELL → STOCK → TYPES → REVIEW; mocked + approved before code per spec §13), and the mandatory motion (one easing curve, card-accept olive micro-interaction, count-up totals, ~250ms step x-slide, prefers-reduced-motion). No commit, no agent network calls, no schema work — those are later phases. This phase makes the canvas real, deterministic, testable, and resumable, with all sources feeding ONE staging surface so the operator never sees "products vs catalog items."
3. **[Deterministic Sources — Source Picker, Manual Entry, CSV/XLSX Mapper, Trade Templates](2026-06-13-catalog-setup-wizard/02-deterministic-sources-source-picker-manual-entry-csv-xlsx-mapper-trade-templates.md)** — Phase 2 builds the four deterministic source lanes that feed the shared staging canvas (built in Phase 1): the "How do you want to start?" source picker, manual entry (reusing catalog quick-add patterns), the deterministic CSV/XLSX mapper (a faithful TypeScript port of the iOS CatalogCSVMapper + ProductsCSVMapper — family grouping, header alias auto-map, name→category/unit resolution against company vocabulary, permissive numeric parse), and per-trade template seed data mirroring the industry-presets shape. Every mapper is a PURE function producing Phase-1 StagingCard[] — built strictly test-first with real fixture rows. XLSX is parsed client-side via SheetJS (xlsx) into rows, then funneled through the identical mapper. Uploads auto-route: clean CSV/XLSX → deterministic mapper; messy doc/photo → the agent lane (Phase 4 — wired as a cross-ref boundary here, not implemented). The agent/QB lanes appear in the picker but are stubbed handoffs to Phases 3-4. This phase writes zero rows to the DB and adds zero schema — it only produces staging cards the canvas already renders and the Phase-1 commit (catalog_setup_save) already serializes.
4. **[The Commit Pipeline — catalog_setup_save, vocab pre-resolution, show-diff dedupe, completion](2026-06-13-catalog-setup-wizard/03-the-commit-pipeline-catalog-setup-save-vocab-pre-resolution-show-diff-dedupe-completion.md)** — Phase 3 builds the server-side commit path that turns the accepted staging-card set into real catalog rows via the merge-capable, idempotent `catalog_setup_save` RPC, plus the three pure libraries that feed it. Four pillars: (1) a PURE payload BUILDER (heavy TDD) mapping staging cards → the exact `catalog_setup_save` payload — products + nested options/values + `add_flat` pricing_modifiers for tiers (NEVER `tiered_pricing`) + variant-pinned `product_materials` recipes + single-`family` `catalog_items`/`catalog_variants` + `product_bundle_items`, all with client_id refs so the RPC UPSERTs by id-map rather than double-creating; (2) VOCABULARY PRE-RESOLUTION that auto-creates missing `catalog_categories`/`catalog_units` (via the existing services) and rewrites card refs to real UUIDs BEFORE commit, so the RPC never rejects unknown ids; (3) a PURE SHOW-DIFF dedupe matcher (TDD) matching each card against the live catalog on `lower(trim(sku))` (name fallback when SKU absent) → per-field diff cards (default action show-diff; merge-all + skip) and stamping `external_source`/`external_id`; (4) the API route wiring token→verifyAuthToken→findUserByAuth→**accessToken-bearing client** (NOT service-role — the RPC is SECURITY INVOKER and its scope guard reads `auth.jwt()->>'email'`)→`catalog_setup_save` with a stable session idempotency key, then on success stamps `company_settings.catalog_setup_completed_at`, fires a Sonner toast (client) + a header-rail notification by DIRECT INSERT into `notifications` (the dispatch route self-filters the actor and would no-op), adding a `catalog_ready` NotificationType (additive). CRITICAL VERIFIED CONSTRAINTS surfaced to execution: the RPC writes exactly ONE catalog `family` per call (no top-level `catalog_items` array) → the STOCK module commits one-family-per-call or groups into one family; the RPC is SECURITY INVOKER with a JWT-email company-scope guard so service-role calls are rejected; the additive columns `products.external_source/external_id`, `catalog_items/catalog_variants.external_*`, and `company_settings.catalog_setup_completed_at` DO NOT YET EXIST on prod (the dedupe stamping + completion stamp depend on the Phase-1 additive migration landing first); `company_settings.company_id` is TEXT not uuid.
5. **[The Always-On Setup Agent (+ Phase C autonomous layer)](2026-06-13-catalog-setup-wizard/04-the-always-on-setup-agent-phase-c-autonomous-layer.md)** — Adds the dedicated, narrow, suggest-only Setup Agent — ungated for every company — that streams schema-constrained catalog proposals into the SAME staging canvas the deterministic sources feed (Phase 1 card model). Three capabilities: (1) conversational generate from a description, (2) ENRICH imported/CSV rows with options/tiers/recipes/types that import can't express, (3) DOCUMENT/PHOTO/PASTE extraction (Claude vision/document input) → proposals. Every proposal passes a PURE, TDD'd validator constrained to OPS catalog schemas before it can become an accept/edit/reject card; the agent never writes — the owner approves every row, and the commit still goes through Phase 1's catalog_setup_save. Server route streams structured proposals via the AI SDK (ai + @ai-sdk/anthropic) over Claude (default claude-opus-4-8, adaptive thinking). Auth follows the repo pattern (verifyAdminAuth → findUserByAuth → service-role); permission gate is granular catalog RBAC (catalog.run_setup from Phase 0, never role). Copy is "guided setup" / describe-the-behavior — never "AI" in product (ops-copywriter), audience-language rules honored. Graceful degradation: offline / declined / model-failure falls back to the deterministic survey + template + manual with ZERO data loss — every already-accepted card stays on the canvas. Phase C AUTONOMOUS layer is gated (flagsReady && canAccessFeature('phase_c') client; AdminFeatureOverrideService.isAIFeatureEnabled(companyId,'phase_c') server, fail-closed): it pre-stages a whole proposed catalog and routes through the existing agent_actions approval-queue-service. All AI deps and a new env var (ANTHROPIC_API_KEY) are added in Task 4.1.
6. **[QuickBooks Import — reconcile the read-only pull engine, map QB Items to the catalog, connect-detect entry, plaintext-token gate](2026-06-13-catalog-setup-wizard/05-quickbooks-import-reconcile-the-read-only-pull-engine-map-qb-items-to-the-catalog-connect-detect-entry-plaintext-token-gate.md)** — Phase 5 brings the existing structured-pull source online for the Catalog Setup Wizard's "import from your accounting software" lane.
